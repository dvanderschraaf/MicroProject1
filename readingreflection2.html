Reading and Response: Nielsen Norman Group “10 Usability Heuristics for User Interface Design” - 04/06/26

1. Norman + Nielsen

I think that the most similar heuristic to Norman's concerns is “Visibility of system status”. Norman talks about the importance of making the system's internal state understandable to the user; clear communication like I talked about in the last reading response. This heuristic directly reflects this idea. Both of them argue that it shouldn't be up to the user to have to guess what's happening. This almost mirrors Normans idea of “The Gulf of Evaluation", where the user struggles to understand the systems state.

Another overlap I see is the emphasis on reducing cognitive effort. Norman talks about how the user should not have to remember or infer too much. Nielsen's heuristics emphasizes recognition over recall. Instead of having the user have to learn everything through trial and error, making the system clear through clear communication makes the system a lot more ideal and well built.

2. Challenge Your Own Work

One heuristic that my micro project struggles with is the “match between system and the real world”. My energy system controls the squares behavior but is not something familiar to the user, so it is an abstract concept. When the square moves through states, the reasoning behind why isn't clear to the user. I would say the most clear example of this is when the square collapses and then has to recover. Stop, pause and then moves fast. Without knowing why this happens, instead of it feeling meaningful and on purpose, it would feel random and unintentional. This matters because it weakens the users understanding of the system, making the experience feel less predictable and harder to learn.

A second heuristic that my project struggles with is “user control and freedom”. Once you get caught by the square, the game ends immediately. The only option to restart the game is pressing the space bar. There is no way to recover, undo or avoid failure once you make contact with the square. When this happens, the system instantly removes all control from the player. This is, if not normal, a somewhat familiar system that is common in a lot of games, though, it still represents the hard constraint that limits control from the player. This matters because it creates an abrupt and unforgiving feeling when the system switches from playing to not playing without any intermediate states.

3. Sketchbook Interface Audit (Preliminary)

My sketchbook most clearly violates “consistency and standards”. Navigation between sketches is clear, but very minimal and could be structured a lot better, not to mention, lacking aesthetics. There are no navigation buttons once navigated to one of the contents, so you have to press the back button. There is no consistent visual or structural system guiding the movement between sketches. This creates an unnecessary friction, as the user has to figure out how to backwards navigate. The sketchbook is also lacking information, system status is not always visible, and unclear information on how each sketch relates to the project and probably confusing explanations on each page. This makes navigation difficult for users and can confuse users on where they are, which increases cognitive load.

4. The Hardest Heuristic

The hardest heuristic to implement honestly is “visibility of system status”. It seems straight forward but it becomes difficult when systems get increasingly more complex. In my project, the variables are all interacting together at once. Communicating all of these clearly without overwhelming the player is challenging. It's the concept that I find difficult. It is stressful having to keep the interface simple while making the system fully understandable. Too much information can be overwhelming and too little information can make the player confused. Balancing this concept of clarity and simplicity makes this heuristic the most difficult to achieve in my opinion.
